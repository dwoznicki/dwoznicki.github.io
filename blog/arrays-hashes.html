Arrays vs. Hashes

This week, my cohort and I took our first foray into a back-end programming language, ruby! Ruby is a fairly high-level language, meaning it reads closer to English than low-level coding languages. This makes it easier to understand, and perhaps a good starting point for a coding novice.

In any case, this week we'll be looking into two types of objects in ruby: arrays and hashes. Let's start with the most basic question, "What are these two things?"

What is an array?

This is an array:

my_array = [7, 20, "Ken", "leopard gecko", true]

Perhaps you can tell by looking, but an array is simply a list of objects. The list can be as long or as short as you want. My array above holds five objects. An array can also contain anything you want: integers, strings, booleans, or even other arrays. So you could make an array inside of an array inside of an array! It all gets very meta. One key property of arrays to note is that the position, or sequence, of items is important. But more on that in a bit. First, let's talk about hashes.

What is a hash?

A hash looks something like this:


        my_hash = {
          month: 7,
          day: 20,
          name: "Ken",
          type: "leopard gecko",
          silly?: true
        }

Look familiar? Perhaps my strange list of objects in my_array makes a bit more sense now. Like an array, a hash is a list. But where an array is just a simple list, a hash is a list of key-value pairs. In my hash, I've marked the keys as categories (month, name, etc.) and the values as attributes of my pet (7, "Ken", etc.). A hash will connnect a value to a key and allow you to access the value using that key. Again, the keys and values can be any kind of object your heart desires. I have chosen to use an object in ruby called a symbol for my keys. A ruby symbol (written as either :symbol or symbol:) is somewhat like a string that ruby can process faster than other objects. This makes it a great choice for keys, especially when your hash gets long and ornery.

So, what's the difference between an array and a hash?

As we've noted, both arrays and hashes are essentially lists of information. They do very similar things, but the difference is in how you access them.

As I mentioned above, the order of items in your array is important. That's because the way to access certain elements is to call their spot in the array. Each spot in the array is associated with a number value, or index. The index starts at 0 and goes up by one for each element in the array. So our array from before would have five index spots: 0, 1, 2, 3, and 4. For example, if I wanted to know what object was in the first index spot, I'd use

my_array[0]

and ruby would return

> 7

since that's the first item in my_array. For the second object, I'd write my_array[1], and so on.

Unlike an array, a hash does not particularly care about the order of objects. This is because to find a value in a hash, you'd call the key it's connected to. So let's say I wanted to know the name of my pet. With an array, it would be pretty difficult to find that information unless you knew exactly where it was in the list. But with a hash, I can simply write

my_hash[name:]

and I'll get

> Ken

which is the value connected to name: in my_hash. Pretty great, eh? Hashes are like arrays with the added benefit of being able to associate objects with key words, not just an index number. Nevertheless, both arrays and hashes have their advantages and disadvantages when programming.

When to use an array vs. a hash

Since an array is simpler than a hash, it's best used for any kind of basic list. Let's say you wanted to write a program that searched through the Lord of the Rings books, found every instance of the string "Ent", and pushed the page and line number into a list. An array would make much more sense for this sort of project. You could push the information into a hash, but your keys would probably end up looking like instance_1:, instance_2:, etc. If the keys in your hash would just be a glorified number index, you're probably better off with an array.

But when you want something a little more robust than a simple list, hashes really shine. So now that you've found your page and line numbers, you want to write a short description of each Ent that appears in the book and put it in a list for future prosperity. Your description of Quickbeam is going to be awfully difficult to find if you have to memorize what index number it falls under. But if you were to put your descriptions into a hash, you could easily mark each one with a key, say quickbeam:. So if finding the value you want by index number is going to be extremely difficult, you're probably better of using a hash.
